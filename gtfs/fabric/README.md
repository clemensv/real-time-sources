# gtfs Fabric assets

Post-deploy artefacts for the **gtfs** real-time-sources bridge: a Fabric
Map visualisation of any GTFS-Realtime source's static + RT data. The data
plane is handled by the generic `tools/deploy-fabric/deploy-fabric.ps1`
script (ACI + Event Hubs + KQL database + Eventstream into
`_cloudevents_dispatch`). The generic deployer auto-discovers
`gtfs/fabric/post-deploy.ps1` via the common
"`{source}/fabric/post-deploy.ps1`" hook convention and calls it at the
end of a successful deploy.

This map generalises the NYC + GTFS-DE prototype: it is parametric in the
KQL database name, auto-discovers the source's bounding box from
`StopsLatest`, and assumes the canonical schema laid down by
`gtfs/kql/gtfs.kql` (bracketed base tables `[Routes]`, `[Stops]`, ...
plus the `<X>Latest` materialized views).

## Files

| File | Purpose |
| ---- | ------- |
| `post-deploy.ps1` | Hook auto-invoked by `tools/deploy-fabric/deploy-fabric.ps1`. Acquires Fabric + Kusto tokens, then runs `wire_gtfs_map.py` with the workspace/db/cluster IDs from the deployer's `-Context` hashtable. Auto-creates a blank Map item if `GTFS_FABRIC_MAP_ID` is not set. Can also be run standalone. |
| `wire_gtfs_map.py` | Installs ~15 KQL helper functions in the target database (route shapes, vehicle points, vehicle rectangles, hubs, all-stops with geographic stratification, alert overlays, crowding, multi-modal stations, delay heatmap, vehicle freshness, three-band arrival boards) and idempotently wires the corresponding Fabric Map vector layers via `getDefinition` / `updateDefinition`. Auto-centres the basemap on the source's bounding box (1 / 99 percentile of stop coordinates). |

## Schema assumed (freshly-deployed GTFS database)

Tables generated by `avrotize a2k` with **`--qualified-table-names
--namespace GeneralTransitFeed`**:

  * `['GeneralTransitFeed.VehiclePosition']`,
    `['GeneralTransitFeed.TripUpdate']`,
    `['GeneralTransitFeed.Alert']` (CloudEvents-shaped)
  * `['GeneralTransitFeed.Agency']`, `['GeneralTransitFeed.Routes']`,
    `['GeneralTransitFeed.Stops']`, `['GeneralTransitFeed.Trips']`,
    `['GeneralTransitFeed.Shapes']`, plus every other GTFS-Schedule entity
    (`['GeneralTransitFeed.StopTimes']`,
    `['GeneralTransitFeed.Calendar']`,
    `['GeneralTransitFeed.FeedInfo']`...)
  * `['GeneralTransitFeed.VehiclePositionsFlat']`,
    `['GeneralTransitFeed.TripUpdateFlattened']` - hand-authored derived
    flat tables built by the `.alter table policy update` clauses in
    `gtfs.kql`.

Materialized views (one per base table, all `arg_max(___time, *) by
___type, ___source, ___subject`):

  * `['GeneralTransitFeed.RoutesLatest']`,
    `['GeneralTransitFeed.StopsLatest']`,
    `['GeneralTransitFeed.TripsLatest']`,
    `['GeneralTransitFeed.ShapesLatest']`,
    `['GeneralTransitFeed.AlertLatest']`,
    `['GeneralTransitFeed.TripUpdateLatest']`,
    `['GeneralTransitFeed.VehiclePositionLatest']`, and the same `*Latest`
    for every other entity.

The wire script reads the `*Latest` MVs directly so each layer query is
constant-time regardless of how long the source has been ingesting. Only
the streaming feeds (`['GeneralTransitFeed.VehiclePositionsFlat']`,
`['GeneralTransitFeed.TripUpdateFlattened']`,
`['GeneralTransitFeed.Alert']`) are time-bounded with
`___time > ago(lookback)` against the raw table.

The qualified table names prevent collisions when the same Eventhouse
hosts multiple feeders (e.g. when an aggregator merges several
`<source>/kql/<source>.kql` scripts into one database, generic names like
`Routes`, `Stops`, `Trips` would otherwise overlap across feeders).

## Prerequisites

1. The generic per-source bootstrap has already run for `gtfs`:
   ```powershell
   ./tools/deploy-fabric/deploy-fabric.ps1 -Source gtfs `
       -ResourceGroup rg-gtfs -Workspace <ws-guid> -Eventhouse <eh-guid>
   ```
   This applies `gtfs/kql/gtfs.kql` and stands up the rest of the data
   plane.

2. **A blank Fabric Map item exists** in the workspace, or you let the
   hook auto-create one. The Fabric REST API doesn't (yet) expose
   programmatic creation of `Map` items via the typed surface, but the
   generic `items` API accepts `type=Map`, so this hook does that
   transparently when `GTFS_FABRIC_MAP_ID` is empty.

3. The map item ID is exposed to the hook via the `GTFS_FABRIC_MAP_ID`
   environment variable. If absent the hook will create / reuse a Map
   named `gtfs-map` (override with `GTFS_FABRIC_MAP_NAME`).

## Standalone re-wire

After tweaking a layer query or colour ramp in `wire_gtfs_map.py`:

```powershell
$env:GTFS_FABRIC_MAP_ID = "<map-guid>"
./gtfs/fabric/post-deploy.ps1 `
    -WorkspaceId   "<ws-guid>" `
    -KqlDatabaseId "<kqldb-guid>" `
    -KustoUri      "https://trd-xxxxxxxx.z1.kusto.fabric.microsoft.com" `
    -KustoDatabase "gtfs"
```

`FABRIC_TOKEN` and `KUSTO_TOKEN` will be acquired via `az account
get-access-token` automatically if not set in the environment.

## Why both `seriesGroup` AND a `match` expression?

Empirically, Fabric Maps' documented `enableSeriesGroup` + `seriesGroup`
+ `customColors` schema populates the Data Layers panel legend correctly
but does **not** drive per-feature paint colour - all bubbles/lines fall
back to the static fallback `color` (typically grey). What actually
renders is a MapLibre-style `["match", ["get", column], v1, c1, ...,
fallback]` expression on the paint property.

The wire script writes **both** at deploy time: `customColors` (for the
panel legend) and the `match` expression (for the renderer). The
distinct values are enumerated from the live query result so the palette
stays in sync with the data - if the source publishes a route with a new
hex colour, the legend and rendering pick it up automatically.

## LineString geometry caveat

Kusto's `make_list(pack_array(lon, lat))` **flattens** the inner arrays
to a single 1-D list `[lon, lat, lon, lat, ...]`, which is not a valid
GeoJSON LineString. So the `shape_lines` and `route_alert_lines`
functions build the coordinate strings explicitly and use
`parse_json(strcat(...))` to construct the GeoJSON. Don't try to
"optimize" that to `bag_pack("type","LineString","coordinates",
make_list(pack_array(...)))` - it produces malformed geometry that the
Fabric Map renderer silently drops with a `booleanValid` warning in the
browser console.
