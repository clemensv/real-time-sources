One-Paste Cloud Shell Launcher

The web UI collects the small amount of tenant-specific context we cannot know in advance, then generates a short command for Azure Cloud Shell. Cloud Shell becomes the orchestration runtime; ACI hosts the bridge; Fabric hosts Eventstream, Eventhouse, and the KQL database.

1. Ask only setup-critical questions

Workspace name or id, Azure subscription, resource group, region, source choice, and whether the user wants a new or existing Event Hub namespace.

2. Generate a short-lived launcher

The page creates a signed payload URL so the pasted command stays short while the detailed per-source configuration lives in JSON, not in the clipboard.

3. Paste into shell.azure.com

Cloud Shell resolves Azure and Fabric context, deploys the ACI template, creates the Fabric items, applies KQL, and emits a setup pack plus validation queries.

4. Fall back only when needed

If Eventstream still needs a connection-specific id or mapping name, the launcher pauses with exact paste values instead of leaving the operator to reverse-engineer them.

Invoke-RestMethod 'https://setup.example/launch.ps1?payloadUrl=https%3A%2F%2Fsetup.example%2Fp%2Fbluesky-fabric-7f3b.json' | Invoke-Expression
Why the payload URL matters
It keeps the user-visible command stable while allowing the backend to emit source-specific ACI parameters, KQL bundles, Eventstream definitions, and validation queries.
What runs where
Cloud Shell orchestrates. Azure Container Instances run the feeder. Fabric stores and routes the stream. The browser only gathers inputs and renders status.
Why not embed Cloud Shell
The supported experience is to send people to Azure surfaces such as shell.azure.com or the Azure portal, not to host a general-purpose Cloud Shell terminal inside an arbitrary site.
Why this is feasible now
  • Fabric APIs can list workspaces so the launcher can validate the target workspace early.
  • Fabric can create Eventhouse and KQL Database items, so schema bootstrap can become scripted instead of doc-driven.
  • Eventstream creation also has an API and definition format, so topology generation is no longer just a design sketch.
What the repo already has

ACI coverage is the strongest current asset because most bridges already ship Azure deployment templates. KQL coverage exists but is still incomplete, which is why the mockup assumes generated bundles per bridge.

Main open integration risk

Eventstream definitions still need connection-level values such as connection names, mapping rule names, or data connection ids. If those cannot be derived or created cleanly by API, the launcher should degrade into an exact guided handoff instead of pretending full automation exists.
